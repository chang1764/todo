# 03 — 시나리오

## 시나리오 1: 박준호 (PM) - 아침 스탠드업 전 진행 상황 체크

**Context**  
월요일 오전 9시, 스탠드업 미팅 5분 전

**Flow**
1. 모바일로 보드 열기
2. 각 컬럼 카드 수 훑어보기 (To Do / In Progress / Review / Done)
3. 기한 오늘인 카드 확인
4. 우선도 높은 미완료 카드 체크
5. 팀 대시보드: "이번 주 완료 3개, 진행 중 5개, 이수진 가장 활발" 파악
6. "오늘 Done 예상은 3개, In Progress는 5개인데 하나 병목" 판단

**Insight**  
- 빠른 열람이 중요 (3분 이내)
- 기한과 우선도 시각적으로 눈에 띄어야 함
- 팀 대시보드로 팀 상황 한눈에 파악

---

## 시나리오 2: 박준호 (PM) - 새 기능 요청 받아 보드에 추가

**Context**  
월요일 오전, 마케팅팀으로부터 "긴급 기능 추가 필요" 보고

**Flow**
1. 보드 열기 (데스크톱)
2. "새 카드 추가" 클릭
3. 제목만 입력: "사용자 분석 대시보드"
4. 우선도는 나중에 설정하기로 (일단 default)
5. 기한도 나중에 수정 ("이번 금요일"로 설정)
6. "To Do" 컬럼에 저장
7. 실시간으로 이수진, 김다현 화면에 카드 출현

**Insight**  
- 카드 생성 진입장벽이 낮아야 함 (최소 제목만 필요)
- 우선도/기한은 선택사항
- Realtime 동기화로 팀이 즉시 인지

---

## 시나리오 3: 박준호 (PM) - 우선도 급변경

**Context**  
화요일 오전, "금요일 deadline" 카드를 발견 → 오늘이 수요일

**Flow**
1. 데스크톱 보드에서 카드 클릭
2. 우선도 "medium" → "high"로 변경
3. 담당 개발자 이수진이 자동으로 보드 업데이트 목격 (Realtime)
4. 이수진 모바일에서 카드가 빨강으로 강조됨
5. 우선도 변경이 즉시 반영

**Insight**  
- 우선도 변경이 자주 일어남 (빠른 업데이트 필수)
- Realtime 동기화로 팀이 즉시 신호 받음
- 개발자가 우선도 색상으로 영향도 인지

---

## 시나리오 4: 이수진 (개발자) - 개발 시작, 카드 "In Progress"로 이동

**Context**  
수요일 오전, 칸반 보드의 "사용자 분석 대시보드" 카드 할당받음

**Flow**
1. 모바일로 보드 열기
2. 자신의 카드 찾기 ("사용자 분석 대시보드")
3. 카드를 "To Do" → "In Progress"로 드래그
4. 카드가 스르르 옮겨짐 (Realtime)
5. 대시보드에서 "진행 중" 개수 증가 (5 → 6)
6. PM 박준호가 모바일에서 즉시 확인

**Insight**  
- 모바일에서 드래그가 부드러워야 함
- 개발자는 자신의 카드 중심으로만 봄
- Realtime으로 PM이 진행 상황 즉시 인지

---

## 시나리오 5: 이수진 (개발자) - 개발 완료, 코드 리뷰 대기

**Context**  
목요일 오후, 개발 완료 후 코드 리뷰 진행 중 → 리뷰 통과

**Flow**
1. 모바일로 "In Progress" 카드 클릭
2. 모달 열림: 제목, 설명, 우선도, 기한
3. 댓글 추가 (선택): "개발 완료, 리뷰 요청합니다"
4. 카드를 "In Progress" → "Review"로 드래그
5. 활동 로그 자동 기록: "이수진이 In Progress → Review로 이동 (15:30)"
6. PM과 디자이너가 카드가 Review로 옮겨진 것 확인

**Insight**  
- 댓글은 선택사항이지만, 있으면 협업 강화
- 활동 로그로 자동 기록되어 투명성 증대
- Realtime으로 팀이 즉시 진행 상황 파악

---

## 시나리오 6: 김다현 (디자이너) - 리뷰 요청 댓글 확인

**Context**  
목요일 오후, 협업 Slack에서 알림 받음 (또는 보드에서 직접 확인)

**Flow**
1. 주간에 3회 정도 보드 확인 (금요일)
2. "Review" 컬럼에 "사용자 분석 대시보드" 카드 발견
3. 카드 클릭, 댓글 확인: "개발 완료, 리뷰 요청합니다"
4. 자신의 할 일 추가: "UI 리뷰 완료했습니다"
5. 모바일에서 기한 "금요일"을 확인

**Insight**  
- 디자이너는 "Review" 컬럼 중심으로 봄
- 기한이 가장 중요한 신호
- 댓글로 협업 흐름 추적

---

## 시나리오 7: 박준호 (PM) - 주간 리뷰 (통계)

**Context**  
금요일 오후, 주간 회의 10분 전

**Flow**
1. 데스크톱 보드 열기
2. 팀 대시보드 확인:
   - 이번 주 완료: 8개
   - 진행 중: 2개 (대부분 완료 상태)
   - 가장 활발: 👨‍💻 이수진 (6개 완료)
   - 기한 임박: 0개 (성공!)
3. "Done" 컬럼에 쌓인 카드 한눈에 (색상, 기한)
4. 팀장 회의에서 보고: "이번 주 목표 120% 달성, 이수진 MVP"

**Insight**  
- 통계를 한눈에 볼 수 있어야 함
- 팀 대시보드로 팀원 기여도 가시화
- 기한 초과 0 = 팀 협력 우수

---

## 설계 원칙

| 원칙 | 설명 | 페르소나 |
|---|---|---|
| **P1: 빠른 진입** | 카드 추가/수정이 3클릭 이내 | 박준호, 이수진, 김다현 |
| **P2: 실시간 동기화** | 한 사람의 변경이 즉시 다른 사람 화면에 | 전체 팀 |
| **P3: 우선도 시각화** | 우선도가 색/아이콘으로 눈에 띄게 | 박준호 (우선도 자주 변경) |
| **P4: 기한 강조** | 기한 임박/초과 카드 빨강 경고 | 박준호, 김다현 (deadline 중시) |
| **P5: 모바일 최적** | 모바일에서도 드래그 정확하고 빨라야 함 | 이수진, 김다현 (모바일 80% 사용) |
| **P6: 선택적 댓글** | 댓글은 있으면 좋지만 필수 아님 | 이수진 (번거로움 피함) |
| **P7: 통계 한눈에** | 컬럼별 카드 개수, 완료율 가시화 | 박준호 (주간 리뷰) |
| **P8: 활동 로그** | 카드 변경 이력이 자동 기록됨 | 박준호, 전체 팀 (추적) |
| **P9: 개발자 드래그** | 드래그 & 드롭 중심 (모바일 터치 최적) | 이수진 (개발자, 자주 사용) |
| **P10: 팀 대시보드** | 팀원 활발도, 완료율, 팀 분위기 공유 | 박준호 (통계 필수) |

## 페르소나별 핵심 사용 흐름

### PM (박준호) 일일 루틴
```
아침 → 스탠드업 전 통계 확인 (3분)
    → 우선도 조정 (수시)
    → 주간: 금요일 통계 리뷰 (10분)
```

### 개발자 (이수진) 일일 루틴
```
아침/점심/저녁 → 자신의 카드 상태 변경 (1분씩, 3회)
    → 개발 진행 상황을 드래그로 표현
    → 완료 시 애니메이션으로 만족감
```

### 디자이너 (김다현) 주간 루틈
```
월/수/금 → "Review" 컬럼 확인 (3분)
    → 기한 확인, 댓글로 피드백
    → 기한 임박 카드 우선 확인
```
